# Risk-Aware Portfolio Allocation Dashboard

This project builds a decision-support system for a risk-aware investor. It forecasts near-term market risk and adjusts portfolio exposure between SPY (equity risk) and SHY (short-duration Treasuries) to test whether dynamic de-risking can improve risk-adjusted outcomes versus static allocations. The core question is simple: **can a forward-risk signal improve downside control without abandoning market participation?**

## Problem Statement

Static portfolios do not adjust to changing market risk. This project tests whether a forward-risk signal can improve downside control by dynamically reallocating between SPY and SHY.

## Investor Objective

The system is designed for a risk-aware long-term investor who wants better downside control than a static benchmark allocation.

## Project Approach

Pipeline flow (in order):
1. Data ingestion and validation (`src/data_loader.py`, `src/validation.py`)
2. Leakage-safe feature engineering (`src/features.py`)
3. Forward 5-day realized volatility label (`src/labels.py`)
4. Walk-forward, out-of-sample model training (`src/split.py`, `src/model.py`)
5. Prediction-to-weight decision rule (`src/strategy.py`)
6. Backtesting with transaction costs and static benchmarks (`src/backtest.py`)
7. Threshold sensitivity analysis as robustness analysis (`src/sensitivity.py`)

## Strategy Logic

Decision rule:
- Predicted volatility **above** threshold → defensive allocation
- Predicted volatility **below** threshold → risk-on allocation

Default allocations:
- Defensive: **30% SPY / 70% SHY**
- Risk-on: **80% SPY / 20% SHY**

## Benchmarks

The dynamic strategy is compared against:
- Static **80/20 SPY/SHY**
- Static **60/40 SPY/SHY**

## Backtest Design

Key design choices:
- **Next-day weight application**: decisions at date `t` apply from the next return date
- Transaction costs are deducted on effective rebalance dates using turnover
- No same-day look-ahead in allocation application
- Model predictions are generated walk-forward and strictly out-of-sample

## Threshold Sensitivity (Robustness Analysis)

Threshold sensitivity is used to test robustness, not to “mine” a single best threshold. It helps evaluate trade-offs across plausible threshold values:
- Sharpe ratio
- Max drawdown
- Turnover
- Transaction costs

## Results

This repository does not hardcode one canonical “final” metric snapshot. Instead, results are generated by running the dashboard and notebook pipeline with current configuration and data. The app and notebook report dynamic strategy outcomes alongside static benchmark comparisons and threshold robustness tables/charts.

## Interpretation

How to read outcomes:
- The strategy may help when forward-risk forecasts identify periods where de-risking reduces deeper drawdowns.
- The strategy may lag in fast upside recoveries if it stays defensive too long.
- There is a trade-off between drawdown control and upside participation.
- Robustness across thresholds matters more than any one point estimate.

## Limitations

- Volatility predictability is unstable across regimes.
- Threshold choice affects behavior and performance.
- SPY/SHY is a simplified two-asset universe.
- Backtests rely on historical structure and do not guarantee future results.

## Setup Instructions

From repository root:

```bash
python -m venv .venv
source .venv/bin/activate  # Windows: .venv\\Scripts\\activate
pip install -r risk_aware_allocation/requirements.txt
```

Run tests:

```bash
pytest risk_aware_allocation/tests -q
```

Launch Streamlit app:

```bash
streamlit run risk_aware_allocation/app/streamlit_app.py
```

Open notebook:

```bash
jupyter notebook risk_aware_allocation/notebooks/01_project_walkthrough.ipynb
```

## Usage Instructions

1. Run the Streamlit dashboard.
2. Select model and strategy controls in the sidebar.
3. Review main performance table versus 80/20 and 60/40 benchmarks.
4. Inspect equity, drawdown, weights, and predicted-vs-actual volatility charts.
5. Use threshold sensitivity charts to judge robustness trade-offs (not to overfit one threshold).
